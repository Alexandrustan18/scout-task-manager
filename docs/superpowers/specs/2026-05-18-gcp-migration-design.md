# GCP Migration Design — Scout Task Manager off Supabase

**Status:** Approved (2026-05-18)
**Author:** Bogdan + Claude
**Decision drivers:** Supabase organization exceeded 5.5 GB/month egress (consumed 11.97 GB), service blocked until June 15. User chose to migrate to existing GCP infrastructure rather than upgrade to Supabase Pro.

## Goals

1. Restore platform availability for the 10-user team on GCP infrastructure.
2. Self-host the full Supabase stack so the app keeps working with **zero behavioral changes** beyond the API URL + anon key.
3. Migrate all production data with zero loss (3,000 task rows + 17 config blobs + backup rows).
4. Stay within ~$25/month operational cost.

## Non-goals

- Refactoring the app to use a different API surface (PostgREST stays).
- Replacing realtime with polling.
- Adding new features.
- Multi-region or HA — single VM is fine for 10 users.

## Constraints

- **Team is OK with downtime** during the migration (user chose Option B: one focused session, no Supabase Pro stopgap). Realistic offline window: 4-6 hours.
- **Supabase REST is blocked** (HTTP 402 `exceed_egress_quota`) — the migration must extract data via the Supabase **dashboard SQL Editor**, which remains accessible during quota throttling.
- Existing GCP project `scout-ai-491712` is the deploy target. Region `europe-west1-b` to match existing VMs (`scout-ai`, `scout-ai-v2`).

## Architecture

### Infrastructure

| Resource | Spec | Rationale |
|---|---|---|
| VM | `scout-tasks` — `e2-medium`, Debian 12, 50 GB `pd-balanced` boot disk, static external IP, zone `europe-west1-b` | e2-medium = 4 GB RAM, comfortable for the Supabase stack (~2.5 GB working set). ~$18/mo before disk. |
| Network | `default` VPC, `default` subnetwork | Matches existing VM convention. |
| Network tags | `http-server`, `https-server`, `scout-tasks-api` | First two pre-existing for the `allow-http` rule; third is for a new tcp 80/443 rule. |
| Firewall | New rule `allow-tasks-https` — INGRESS tcp:80,443 from 0.0.0.0/0 to tag `scout-tasks-api` | Required because existing `allow-http` only covers custom ports 8000/3001/8001/3002, not the standard 80/443 we'll use behind nginx. |
| Service account | `scout-tasks-sa@scout-ai-491712.iam.gserviceaccount.com` with `storage.objectAdmin` on the backup bucket | Dedicated identity for backup uploads. Minimal blast radius vs reusing `scout-ai-sa`. |
| DNS | `api.work-heyads.ro` A record → VM static IP | Subdomain isolation; main `work-heyads.ro` stays on Vercel. |
| Backup bucket | `gs://scout-ai-491712-tasks-backups`, lifecycle: delete > 30 days | Nightly `pg_dump` archives. |
| Estimated monthly cost | **~$25** | VM ~$18 + disk ~$5 + egress ~$1 + bucket ~$1 |

### Software stack (on the VM, all via docker-compose)

| Container | Purpose | Port |
|---|---|---|
| `postgres` (Postgres 15) | Source of truth. JSONB schema same as Supabase Cloud. | 5432 (internal only) |
| `postgrest` | REST API the app already speaks (`/rest/v1/app_data?...`). | 3000 → nginx |
| `realtime` (Supabase Realtime, Phoenix/Elixir) | Broadcasts `postgres_changes` to the app's WebSocket subscriptions. | 4000 → nginx |
| `gotrue` | Bundled by Supabase self-host. Unused by app (we use the publishable anon key only), but kept because the official compose file expects it. | 9999 (internal) |
| `kong` (or nginx) | Reverse proxy + TLS termination + path routing. | 80, 443 (public) |
| `studio` (admin UI) | Web UI for the database. **Not exposed publicly** — accessed via SSH tunnel only. | 3000 (internal) |

We will use the **official Supabase self-host docker-compose** (`github.com/supabase/supabase/tree/master/docker`) rather than hand-rolling. This is the same image set Supabase Cloud runs, which means the REST + Realtime API surface is byte-for-byte identical to what the app currently calls. **No app-side behavioral changes** other than:

- 4 hardcoded URLs `https://ploucecgizjwyumzmhmo.supabase.co` → `https://api.work-heyads.ro`
- 1 publishable key → new anon key generated by the self-host setup script

### Data flow during migration

```
Supabase Cloud (blocked, but dashboard SQL Editor works)
    │
    │  pg_dump via SQL Editor → CSV per row
    │  OR direct connection string (if Supabase grants it during throttle)
    ▼
Local files: /tmp/migration/<rowname>.csv
    │
    │  scp to VM (rsync optional if files are large)
    ▼
GCP VM scout-tasks
    │
    │  psql \copy into app_data table
    ▼
PostgREST + Realtime serve the data
```

### Why not Cloud SQL?

For a 10-user app, Cloud SQL adds:
- ~$25-50/mo extra cost (smallest tier ~$25/mo at lowest CPU)
- VPC network complexity (private IP, proxy sidecar)
- Slightly slower latency (cross-machine vs. localhost)

Tradeoff: managed Postgres backups + monitoring. Replicated cheaply with cron `pg_dump` to GCS. Skip Cloud SQL for now; can move later if scaling demands.

## Cutover plan (high level — full plan in implementation doc)

1. Provision VM via `gcloud compute instances create` (one-shot commands, not Terraform — we're not setting up IaC for this single VM).
2. Bootstrap Docker + Supabase self-host on the VM.
3. Configure nginx + Let's Encrypt cert for `api.work-heyads.ro`.
4. Export data from Supabase dashboard SQL Editor → CSVs.
5. Import to new Postgres (`\copy` per row).
6. Update `App.jsx` (4 URLs + 1 key), commit, push to GitHub.
7. Vercel auto-deploys new frontend.
8. Smoke test via Playwright (admin login → create task → status changes → realtime).
9. Tell team to hard-refresh.
10. Set up nightly backup cron.

Realistic total: 4-6 hours of focused work. Team offline that whole window.

## Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Data export from Supabase fails (dashboard SQL Editor restrictions) | Medium | If SQL Editor blocks exports too, fall back to: pay Supabase Pro for 1 month → unblock REST → run export script → migrate. ~$25 insurance. |
| Realtime self-host config mismatch (Phoenix tokens, JWT secrets) | Medium | Use Supabase's official setup script which generates secrets correctly. Test realtime end-to-end before declaring done. |
| Let's Encrypt rate limits if we iterate too fast | Low | Use Let's Encrypt staging cert during testing, production cert only at cutover. |
| GoTrue dependency surfacing | Low | App doesn't use GoTrue auth — only PostgREST + Realtime. Should be invisible. Confirmed via grep of `App.jsx`. |
| Migration leaves orphan Supabase project costing money | Low | Pause/downgrade Supabase project to Free after migration verified, cancel paid org if applicable. |

## Success criteria

- All 2,981 per-row task entries (`task_*` rows) present in new Postgres, count matches Supabase pre-migration.
- All 17 config row types (`team`, `pipelineRules`, `notifs`, `statusHistory`, `timers`, etc.) present.
- Backup rows (`tasks_backup_*`, `oana_tasks_backup_*`) preserved.
- Playwright end-to-end test passes: admin login → create task → mark Done → verify pipeline auto-spawn → realtime sees the change.
- Production bundle on Vercel updated to point at `https://api.work-heyads.ro`.
- Nightly backup cron writes a fresh dump to `gs://scout-ai-491712-tasks-backups`.
- App functional for all 10 user accounts post-cutover.

## Out of scope (deferred)

- HA / multi-zone deploy.
- Cloud SQL migration.
- Replacing PostgREST with a hand-rolled API.
- Refactoring the `app_data` schema (still single table with `id` + `data` JSONB).
- Migrating the legacy `id=tasks` and `id=tasks_archive` blob rows — they're already obsolete (Phase 3 made them backups), but we'll preserve them in the migration so we don't lose history.
